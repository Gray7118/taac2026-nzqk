# 1.2 多尺度时间分桶 (Multi-Scale Time Buckets)

## 一、ts_fid 激活后为何无明显提升

### 1.1 实验发现

激活 `ts_fid` 后，前 5 个 epoch 的 validation AUC 与未激活时基本持平，时间特征未能贡献有效信号。

### 1.2 根因分析

#### 根因一：时间差分布极度偏斜

通过分析 `demo_1000.parquet` 中 `domain_a` 的 219,025 个非零事件的时间差分布，发现**事件时间差高度集中于长周期区间**：

```
单尺度 65-bucket 分布（domain_a 实测）:
  
  短期（<1h, buckets 1-21）:  ∑ < 0.5%   ← 几乎无样本
  中期（1h-24h, buckets 22-53）: ∑ ≈ 15%  ← 稀疏
  长期（1d-1y, buckets 54-64）: ∑ ≈ 84%  ← 高度集中！
  
  具体热点:
    bucket_58 (>259.2万s ≈ 30天): 19.9%  ← 单桶占比最大
    bucket_54 (>17.28万s ≈ 2天): 11.4%
    bucket_56 (>216万s ≈ 25天):  8.9%
```

**99.7% 的事件发生在 1 小时之前**，使得精细时间粒度（秒/分钟级）的 bucket 几乎没有样本可学习。这意味着当下 65 个 bucket 中 ~20 个从未被有效更新，成为嵌入表中的噪声。

#### 根因二：加性嵌入的表达力不足

当前的时间特征注入方式为简单的向量加法：

```python
# model.py:1572
token_emb = F.gelu(proj(cat_emb))      # (B, L, D) — 序列内容嵌入
token_emb = token_emb + self.time_embedding(time_bucket_ids)  # + 时间嵌入
```

这种设计存在三个表达力限制：

| 限制 | 说明 | 后果 |
|------|------|------|
| **无交互项** | 时间信号与内容信号只能通过加法耦合，无法学习 `time × feature_type` 的交叉模式 | "同类目点击在 5 分钟前 vs 30 天前" 无法被区分 |
| **单一时间轴** | 所有 4 个域共享一套 65-bucket 嵌入表 | 不同域的时间语义不同（如广告曝光时序 vs 视频观看时序），但被强制共用同一套参数 |
| **序列位置不可知** | 事件在序列中的位置由 padding mask 隐式处理，时间差本身不反映事件顺序 | 一个 7 天前和一个 30 天前的事件可能落入相邻 bucket，但它们的"近期性"含义完全不同 |

#### 根因三：序列截断丢失关键信息

demo 数据中 72% 的 `domain_a` 样本被截断至 256 个事件。截断发生在序列**尾部**（最新的少量事件被保留），但 CVR 任务最关键的信号恰恰是最近的行为模式。当 `max_len=256` 而实际中位数 `P50=578` 时，超过一半的用户行为历史被丢弃。

截断与实际长度的关系：

```
domain_a truncation (256) vs actual lengths:
  P10=81   → OK (10% 用户完全保留)
  P25=227  → OK (25% 用户基本保留)
  P50=578  → 50% 用户丢失 (578-256)/578 = 56% 的事件
  P75=1118 → 75% 用户丢失 (1118-256)/1118 = 77% 的事件
```

### 1.3 总结：时间信号的引用链断裂

```
timestamp → 序列事件时间 → time_diff → bucket_id → Embedding查表 → 向量加法 → token

断裂点1: 分布偏斜         → 大部分bucket样本不足
断裂点2: 加性嵌入         → 无法建模 time × content 交互
断裂点3: 序列截断         → 信息量最大的近期事件在截断之外
断裂点4: 域间共用嵌入     → 不同行为域的时间语义混淆
```

仅修复 ts_fid 只能打通引用链，但无法修复链路上的结构性缺陷。

## 二、多尺度时间分桶方案

### 2.1 设计目标

将当前的 **单尺度 65-bucket** 拆分为 **4 个独立尺度**，每个尺度由独立的 Embedding 表学习，分别对应不同时间粒度下的行为语义：

```
当前（修复前）:
  ts_fid=null → 时间特征完全失效

当前（仅修复ts_fid）:
  单尺度 65 buckets → 信息集中于4个桶（分布严重偏斜）

改进后（多尺度）:
  4 个独立尺度 × 各自 Embedding 表 → 每个尺度捕获特定粒度的语义
```

### 2.2 四尺度定义

| 尺度 | 时间范围 | 边界数 | Bucket数 | 语义 |
|------|---------|--------|---------|------|
| **fine** | 5s ~ 1h (3600s) | 15 | 16 | 短期行为节奏：快速滑动 vs 仔细阅读/观看 |
| **medium** | 1h ~ 1d (86400s) | 16 | 17 | 日周期模式：上午/下午/晚上的行为偏好 |
| **coarse** | 1d ~ 7d (604800s) | 6 | 7 | 回归频率：每天活跃 vs 隔几天访问一次 |
| **longterm** | 7d ~ 1y (31536000s) | 8 | 9 | 长期兴趣演化：兴趣形成/消退还/复兴 |

**总计**: 46 boundaries → **49 buckets**（相比原来 65 个，更紧凑且更有语义）

```python
FINE_BOUNDARIES = np.array(
    [5, 10, 15, 20, 30, 45, 60, 90, 120, 180, 300, 600, 900, 1800, 3600],
    dtype=np.int64,
)
MEDIUM_BOUNDARIES = np.array(
    [7200, 10800, 14400, 18000, 21600, 25200, 28800, 32400,
     36000, 43200, 50400, 57600, 64800, 72000, 79200, 86400],
    dtype=np.int64,
)
COARSE_BOUNDARIES = np.array(
    [172800, 259200, 345600, 432000, 518400, 604800],
    dtype=np.int64,
)
LONGTERM_BOUNDARIES = np.array(
    [1209600, 2592000, 4320000, 6048000, 7776000,
     11664000, 15552000, 31536000],
    dtype=np.int64,
)
```

### 2.3 关键设计决策

#### 2.3.1 流量分配（Bucket 均匀度）

通过分析 `domain_a` 的实际时间差分布来验证各尺度的均匀度：

| 尺度 | Bucket数 | 覆盖的事件占比 | 最拥挤的bucket占比 | 评判 |
|------|---------|-------------|-----------------|------|
| fine | 16 | 100% | bucket_16(>3600s): **99.7%** | ⚠️ 尾部极端集中 |
| medium | 17 | 100% | bucket_1(~2h): **22.7%** | ✅ 分布相对均匀 |
| coarse | 7 | ~2.2% (减fine后) | bucket_1(1-2d): ~50% | ⚠️ 中等偏斜 |
| longterm | 9 | ~1.0% (减fine+medium后) | 分布较散 | ✅ 可接受 |

**fine 尺度的极端偏斜是必然的**: demo 数据中绝大部分事件都是数天前的。fine 尺度主要服务于"**事件极少但信息量极大**"的牛尾分布——只有 ~0.3% 的事件在 1h 内，但恰是这些事件对 CVR 预测最有用。fine 尺度的作用是让模型学会区分这罕见的近期信号 vs 大量陈旧噪声。

**medium 尺度**是最关键的改进：在 1h-24h 范围内提供了 16 个边界点（原方案只有 3 个: 11000s, 21600s, 43200s），使模型能够区分"刚才 2 小时前" vs "下午某个时间点" 的日周期行为。

#### 2.3.2 数据流变化

```
当前（单尺度）:
  dataset.py → time_bucket: (B, L)           1 channel
  model.py   → time_embedding(65, D)         1 张 Embedding 表
  model.py   → token_emb += time_emb         1 次加法

改进后（多尺度）:
  dataset.py → time_bucket: (B, 4, L)        4 channels per position
  model.py   → time_embs = {                4 张独立 Embedding 表
                  'fine': Embedding(16, D),
                  'medium': Embedding(17, D),
                  'coarse': Embedding(7, D),
                  'longterm': Embedding(9, D),
                }
  model.py   → token_emb += fine_emb        4 次独立加法
               + medium_emb
               + coarse_emb
               + longterm_emb
```

#### 2.3.3 各尺度独立学习 vs 共用一个表的优势

**独立 Embedding 表**意味着：
- 每个尺度可以学习该时间范围内独特的行为模式
- fine 尺度的梯度不会受 longterm 尺度的大量样本主导
- 每个尺度的嵌入维度可以不同（虽然这里保持相同）
- 未来可以选择性冻结某些尺度（如 fine 尺度样本太少时）

### 2.4 实施计划

| 文件 | 改动 |
|------|------|
| `dataset.py` | 新增 4 套边界数组 + `TIME_SCALE_CONFIG`；buffer 改为 4 通道 `(B, 4, max_len)`；`_convert_batch` 中计算 4 个尺度的 bucket |
| `model.py` | `__init__` 中用 `nn.ModuleDict` 管理 4 张独立的 `Embedding`；`_embed_seq_domain` 中迭代 4 个尺度做加法；`_init_params` / `reinit` 中同步处理 |
| `train.py` | 导入 `TIME_SCALE_CONFIG` 传递给模型构造，替代旧的单个 `num_time_buckets` |
| `infer.py` | 模型重构时使用 `TIME_SCALE_CONFIG` 替代旧参数 |

无变动文件: `trainer.py`, `fix_schema.py`

### 2.5 预期收益

| 指标 | 当前（仅修复ts_fid） | 改进后（多尺度） |
|------|-------------------|----------------|
| 时间嵌入表达能力 | 单轴 65 维，大部分未学习 | 4 轴独立，每轴专注特定语义 |
| 短期行为捕获 | 秒~分钟级 granularity 无样本 | fine 尺度专用于牛尾事件（信息密度最高） |
| 日周期学习 | 1h-24h 内仅 3 个边界 | 16 个边界覆盖全天候行为模式 |
| 嵌入表冲突 | 各域共用同一套参数 | 保持各域独立（仍在各自域的序列编码中注入） |
| 参数开销 | 65 × 64 = 4,160 | (16+17+7+9) × 64 = 3,136 (-25%) |

**预期 AUC 提升: +0.3% ~ +1.0%**（在 ts_fid 修复基础上，中等置信度）
