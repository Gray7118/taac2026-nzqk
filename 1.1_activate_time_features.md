# 1.1 激活序列时间特征 (Activate Sequence Time Features)

## 问题诊断

### 现象

Baseline 代码（`model.py`, `dataset.py`）中已经实现了完整的 **time-diff bucket embedding** 时间特征：
- 65 个时间桶边界（5秒 ~ 1年），`NUM_TIME_BUCKETS = 65`
- `nn.Embedding(65, d_model, padding_idx=0)` 即 `time_embedding` 已创建
- `_embed_seq_domain()` 中已实现 `token_emb = token_emb + self.time_embedding(time_bucket_ids)`

但训练时该时间嵌入**完全不工作**——`seq_time_buckets` 始终为全零矩阵（padding idx = 0）。

### 根因

`schema.json` 中所有 4 个序列域的 `ts_fid` 均配置为 `null`：

```json
// schema.json — 修改前
"seq": {
    "domain_a": { "ts_fid": null, ... },
    "domain_b": { "ts_fid": null, ... },
    "domain_c": { "ts_fid": null, ... },
    "domain_d": { "ts_fid": null, ... }
}
```

`dataset.py:364` 行根据此配置判定时间戳列索引：

```python
ts_ci = self._col_idx.get(f'{prefix}_{ts_fid}') if ts_fid is not None else None
# → ts_ci = None（因为 ts_fid = null）
```

`dataset.py:633` 行跳过整个时间分桶逻辑：

```python
if ts_ci is not None:    # ← 永远为 False
    # ... 时间差计算、分桶、赋值 ...

# 结果: seq_time_buckets = 全零矩阵
```

### 数据确认

通过分析 `demo_1000.parquet`（1000 条样本）确认，4 个序列域各有一个特征的实际语义是 **Unix 时间戳**（秒级），其当前被标记为普通超高基数离散特征，vocab 高达 17.7 亿：

| 域 | 时间戳 FID | 当前 vocab | 实际值范围 | 对应日期 |
|----|-----------|-----------|-----------|---------|
| domain_a | seq_**39** | 1,772,725,488 | [1712419199, 1772725487] | 2024-04-06 ~ 2026-03-05 |
| domain_b | seq_**67** | 1,772,725,643 | [1752645918, 1772725642] | 2025-07-16 ~ 2026-03-05 |
| domain_c | seq_**27** | 1,772,725,681 | [1717589186, 1772725680] | 2024-06-05 ~ 2026-03-05 |
| domain_d | seq_**26** | 1,772,725,621 | [1726916040, 1772725620] | 2024-09-21 ~ 2026-03-05 |

**此 bug 造成的双重损害：**
1. 时间差信号完全丢失（time_embedding 零作用）
2. 17.7 亿词表的 Embedding 被创建并训练，浪费显存且信息扭曲

---

## 修复方案

### 一、新增 `fix_schema.py`

- **路径**: 项目根目录 `/fix_schema.py`
- **功能**: 读取平台提供的 `schema.json`，**自动检测**每个序列域的时间戳特征 fid 并设置 `ts_fid`，输出到新文件
- **特点**: 
  - **不修改原始文件**（平台数据目录可能是只读的），输出至 checkpoint 目录
  - **域名无关**（不依赖 `domain_a`/`domain_b` 等 key 命名），按 vocab 大小自动检测，适配任意平台部署

```
Usage:
    python3 fix_schema.py <输入路径> <输出路径>

示例:
    python3 fix_schema.py ./data/schema.json ./ckpt/schema_patched.json
```

#### 自动检测机制：基于 vocab 阈值

平台 schema 的序列域名可能与本地不匹配（如平台使用 `seq_domain_1` 而非 `domain_a`），因此 `fix_schema.py` 不依赖任何硬编码的域名或 fid 映射，而是利用一个**无法伪造的信号**——时间戳特征在 schema 中登记的 vocab 高达 ~17.7 亿，远超任何正常离散特征：

```python
TIMESTAMP_VOCAB_THRESHOLD = 1_000_000_000

# 对 schema["seq"] 下每个域（无论 key 名是什么）：
#   扫描 features 列表 → 找到 vocab > 1e9 的特征 → 设其为 ts_fid
```

| 判断依据 | 值 |
|---------|-----|
| 时间戳特征 vocab | ~1,772,725,488 |
| 最大非时间戳离散特征 (domain_c seq_47) | ~278,677,640 |
| 检测阈值 | 1,000,000,000 |

中间有 ~1.5B 的 gap，不会误判。若某个域有多个候选（几乎不会发生），取 vocab 最大的那个。

#### 幂等性

脚本会检查当前 `ts_fid` 值，如果已经是正确的则跳过，支持重复执行。

### 二、修改 `run.sh`

在训练启动前插入两步：

```bash
# 1. 打补丁
python3 fix_schema.py $DATA_DIR/schema.json $CKPT_DIR/schema_patched.json

# 2. 传入训练
python3 train.py --schema_path $CKPT_DIR/schema_patched.json ...
```

实际改动位置：在 `export TRAIN_TF_EVENTS_PATH` 之后、GPU 检测之前。

## 修复后的数据流

```
修复前：
  schema.json (ts_fid: null)
    → dataset.py: ts_ci = None
    → seq_time_buckets = 全零矩阵
    → model.py: time_embedding(0) = 零向量
    → 时间特征: ❌ 不工作

修复后：
    schema_patched.json (ts_fid = 由 vocab 自动检测)
      → dataset.py: ts_ci = 正确的列索引
      → 读取时间戳列 → time_diff = timestamp - seq_ts
      → searchsorted(BUCKET_BOUNDARIES, time_diff) → time_bucket_ids
      → model.py: time_embedding(time_bucket_ids) → d_model 向量
      → token_emb += time_emb → 时间特征: ✅ 工作
```

**附带的正面副作用**：时间戳列从 `sideinfo_fids` 中排除（`dataset.py:323-324`），不再创建 17.7 亿词表的 Embedding，**节省显存**。

## 预期效果

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| time_embedding 输入 | 全零 (padding) | 有效 bucket_id [1, 64] |
| 时间语义 | 无 | 每个序列事件距今时间（5s~1年，65 级） |
| 序列 Embedding 表 | 含 1 个 17.7 亿词表的 Embedding | 不含该 Embedding |
| 显存 | 浪费 | 节省 |

**预期 AUC 提升：+0.1% ~ +0.5%**（保守估计，仅启用已存在但未生效的基础时间特征）。
